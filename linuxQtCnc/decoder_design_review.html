二、设计模式合规性审查

审查维度

依据经典设计模式（GoF 23种）及 SOLID 原则逐项审查 decoder 目录下全部代码。

🔴 P0-1：gc_update_modal() — 巨型 if-else 链，违反开闭原则 (OCP) 和单一职责原则 (SRP)

文件：gparse.c:160-300 | 约140行连续 if-else

问题：每新增一个G/M代码支持，必须修改这个巨型函数，违反OCP。G字 (65行) 和M字 (15行) 的所有处理逻辑堆叠在同一个函数中，单一函数承担了"G代码识别 + M代码识别 + S/F/T字处理"三重职责。

影响：代码可读性极差，测试困难，修改风险高。

✅ 整改建议：策略模式 (Strategy) + 查表法

使用静态查找表替代 if-else 链：

// 定义命令处理器函数指针类型
typedef uint8_t (*gcode_handler_t)(uint8_t ch, float value, 
    uint8_t *motion, uint8_t *has_motion, uint8_t *group);

// G代码静态映射表
typedef struct { uint8_t code; uint8_t group; 
                 uint8_t motion; gcode_handler_t handler; } gcode_table_entry_t;

static const gcode_table_entry_t g_code_table[] = {
    {GC_G0,  GC_GROUP_MODAL_G0, GC_G0,  NULL},
    {GC_G1,  GC_GROUP_MODAL_G1, GC_G1,  NULL},
    {GC_G17, GC_GROUP_MODAL_G17_19, 0,  gc_set_plane_handler},
    // ... 
};
// gc_update_modal 缩减为: 查表 → 调用 handler (或无handler直接设置modal)

🔴 P0-2：gc_execute_line() 函数过长 — 违反单一职责原则 (SRP)

文件：gparse.c:302-430 | 约130行

问题：单一函数承担了：词法解析 (while循环)、模态更新、各轴坐标赋值 (X/Y/Z/A/B/C/I/J/K/P/Q/L/D 共14个if分支)、辅助功能执行、子程序处理、宏程序处理、运动指令执行。七种职责混在一个函数中。

✅ 整改建议：模板方法模式 (Template Method) + 命令模式 (Command)

将解析流程拆分为独立阶段，每阶段一个函数：

uint8_t gc_execute_line(uint8_t ch, char *line) {
    // 阶段1: 词法解析 → 填充 parsed_words[]
    error = gc_parse_line_lex(ch, line, &parsed);
    // 阶段2: 语义处理 → 根据parsed_words更新modal和target
    error = gc_process_words(ch, &parsed, &motion, target, offset, flags);
    // 阶段3: 辅助功能
    nck_aux_execute(ch);
    // 阶段4: 子程序/宏程序处理
    error = gc_handle_program_flow(ch, motion);
    // 阶段5: 运动执行
    error = gc_execute_motion_command(ch, motion, target, offset, flags);
    return error;
}

🟡 P1-1：全局状态暴露 — 违反封装原则和迪米特法则

涉及文件：decoder/ 下几乎所有 .c 文件

问题：所有模块通过 chan[ch].dcd.ptr_gc_state->xxx 直接访问其他模块的内部状态，链式调用深度达到 4-5 级。例如 fixedcycle.c 中：
chan[ch].dcd.ptr_gc_state->ptr_modal->modal_g40_42 — 穿透了 channel_t → decoder_t → gc_state_t → gc_modal_t 四层结构。

违反： 迪米特法则 (Law of Demeter)：模块不应知道其"朋友的朋友"的内部结构。
封装原则：各模块应通过接口访问而非直接访问结构体成员。

✅ 整改建议：外观模式 (Facade) + 访问器函数

为 decoder 模块提供统一的上下文访问接口，各子模块只通过接口函数获取所需数据：

// decoder_context.h — decoder 统一上下文接口
float dc_get_current_x(uint8_t ch);
float dc_get_current_z(uint8_t ch);
float dc_get_feed_rate(uint8_t ch);
uint8_t dc_get_modal_g40_42(uint8_t ch);
void dc_set_current_z(uint8_t ch, float z);
// ... 约20-30个访问器函数
// 替代所有 chan[ch].dcd.ptr_gc_state->xxx 模式

🟡 P1-2：gcode.c 和 fixedcycle.c 中运动执行代码重复

文件：gcode.c (gc_execute_motion等) vs fixedcycle.c (fixed_cycle_rapid_move, fixed_cycle_feed_move)

问题：两处都有"构造 mc_motion_t → 填充 target/position/feed_rate → 调用 mc_line → 更新 gc_state.input"的相同代码模式。fixedcycle.c 内部的辅助函数与 gcode.c 的 gc_execute_motion 几乎完全重复，差异仅在于 target 坐标的填充方式。

🟡 P1-3：subprog.c 和 macroprog.c 存在大量结构重复

涉及文件：subprog.c/h vs macroprog.c/h

问题：两个模块实现了几乎相同的设计模式：

  • 管理器结构体 (subprog_manager_t vs macro_manager_t) — 结构高度相似
  • 调用栈帧 (subprog_stack_frame_t vs macro_stack_frame_t) — 字段80%相同
  • 程序注册/查找/注销 — 完全相同的模式，仅结构体名不同
  • execute_next / get_line_at_index — 完全相同的实现逻辑
  • parse_program_number / parse_loop_count — 代码一字不差
  • debug函数 (print_stack/print_programs/print_status) — 相同结构

重复度估算：约 60% 的代码可合并为公共基类/模板。

✅ 整改建议：提取公共抽象基模块

// program_manager.h — 通用程序管理器 (抽象子程序和宏程序的公共部分)

// (子程序的M98/M99映射、宏程序的变量系统和表达式计算)

🟡 P1-4：半径补偿 (radiuscompensation) 和刀尖补偿 (tooltipcomp) 缺少统一接口

涉及文件：radiuscompensation.c/h + tooltipcomp.c/h

问题：两者都实现了刀具补偿功能，但：

🟡 P1-6：错误处理不一致

涉及文件：解码器全部模块

问题：

  • gparse.c/gcode.c 使用 SYS_STATUS_* (0x0000-0x01FF) 错误码
  • gc_error.c 使用 GC_ERROR_* (0x0200-0x03FF)
  • subprog.c 使用 SUBPROG_STATUS_* (0x0400-0x05FF)
  • macroprog.c 使用 MACRO_STATUS_* (0x0600-0x07FF)
  • 但各模块混用：fixedcycle.c 返回 SYS_STATUS_WRONG_CHANNEL_INDEX，error 定义在 gc_error.h 中却很少被实际调用
  • 某些函数返回 uint8_t，某些返回 uint16_t，类型不统一
  • 没有统一的错误传播机制 — 每个调用点都需要手动检查并返回

✅ 整改建议：统一错误码类型 + 错误传播宏

// 统一使用 uint16_t 错误码
typedef uint16_t nck_status_t;

// 错误传播宏 (类似 Rust 的 ? 操作符)
#define NCK_TRY(expr) do { \
    nck_status_t _s = (expr); \
    if (_s != SYS_STATUS_OK) return _s; \
} while(0)

// 使用示例
nck_status_t gc_execute_line(uint8_t ch, char *line) {
    NCK_TRY(gc_parse_line_lex(ch, line, &parsed));
    NCK_TRY(gc_process_words(ch, &parsed, &motion, target, offset, flags));
    // ...
}

🟡 P1-7：gc_parse_word() 存在重复解析

文件：gparse.c:113-135 + gc_execute_line() 内各轴字处理

问题：gc_parse_word() 调用 mc_read_float() 解析了一个 float value，但在 gc_execute_line() 中处理 X/Y/Z/A/B/C/I/J/K/R/P/Q/L/D 字时，又再次调用 mc_read_float() 读取同一个值。这意味着每个坐标字被解析了两次。

// gc_parse_word 中: mc_read_float(line, &char_counter, &value) — 第1次
// gc_execute_line 中各字处理: mc_read_float(line, &char_counter, &value) — 第2次!

✅ 整改建议：一次解析，统一使用

gc_parse_word() 解析后的 value 应该直接存入统一的 parsed_word 结构体，后续处理不再重复调用 mc_read_float()。

🟡 P1-8：所有模块的 init 函数模式重复但无统一接口

涉及文件：所有模块的 init 函数

问题：decoder 目录下有 10+ 个 *_init(ch) 函数，它们都有相似的模式（检测 ch 越界 → memset → 设置默认值），但没有统一的初始化接口。gc_init() 中需要逐个手动调用：

void gc_init() {
    // 手动初始化 gc_state (40行)
    gc_coord_init();       // 全局单次

🟢 P2-2：tooltipcomp 未多通道化

文件：tooltipcomp.c/h

问题：decoder 目录下所有其他模块都实现了多通道（带 uint8_t ch 参数），唯独 tooltipcomp 使用全局单实例 ttpc_state。这破坏了模块间的一致性。

static int16_t user_find_code_slot(uint8_t ch, user_code_entry_t *table, 
    uint8_t max_count, uint16_t code) {
    for (int i = 0; i < max_count; i++) {

    }
    return -1;
}

🟢 P2-4：刚性攻丝中紧急退刀使用了阻塞式 while(1) 循环

文件：rigidtapping.c:286-310

while (1) {
    spindle_count = encoder_get_spindle_count() - ...;
    if (...) break;
    if ((hal_get_tick_ms() - start_tick) >= timeout_ms) break;
    if (encoder_get_rpm_filtered() < 10) break;
    hal_delay_ms(1);  // 阻塞等待
}

问题：紧急退刀是实时安全关键操作，但使用了阻塞 while 循环 + hal_delay_ms(1)。在嵌入式系统中，这会阻塞整个主循环，无法响应其他通道或紧急停止信号。应改为状态机驱动。

✅ 整改建议：将紧急退刀改为状态机

紧急退刀不应阻塞等待，而应像 rigid_tapping_execute() 一样每次调用执行一步，由调度器周期调用直到完成。
